How I Built RiskLens

An insurance underwriting and risk scoring platform for SME submissions. Submit a business's risk profile, get a score, a decision (Approved / Referred / Declined), a rule-by-rule explanation of how the score was reached, and a premium estimate. Portfolio dashboard included.

Try it live

The Idea

Having built two insurance platforms at Azur Technology - a property underwriting system and a financial lines platform - I know how underwriting decisions actually get made in practice: a mix of gut feel, spreadsheets, and rule books that live in someone's head. The question I wanted to answer was: what does a modern, explainable underwriting tool look like?

The hardest part of insurance underwriting isn't the maths - it's making the decision auditable. A rate can be justified. A referral needs a reason. RiskLens makes the scoring logic transparent: every rule that touched the submission is shown, with its exact adjustment and the reasoning behind it. That's the thing most legacy underwriting systems don't do well.

Tech Stack

This is the only project in this portfolio with a Python backend. FastAPI was the right choice here - the scoring engine is pure Python logic, and FastAPI's automatic Swagger docs made it easy to test the API directly without building a separate client first.

The Scoring Engine

The engine is a pipeline of independent rules, each returning an adjustment and a plain-English explanation. The final score is the sum of all adjustments, clamped to 0-100. Every rule is a separate function - adding, removing, or reweighting a factor means changing one place in the codebase.

RuleAdjustmentLogic
Base Score+50Starting point for all submissions
Industry0 to +20Technology (0) to Construction (+20); 7 sectors
Claims History-10 to +25Clean record (-10) to 5+ claims (+25)
Revenue-10 to +5>5M revenue (-10); under 500K (+5)
Employee Count-5 to +10Micro (<10 staff, -5) to large workforce (+10)
Location Risk-5 to +20Low-risk area (-5) to very high risk (+20)

Score thresholds map to three decisions: 0-40 Approved, 41-70 Referred, 71-100 Declined. Within that, four rating bands (low, moderate, high, very high) feed the premium calculator independently of the decision outcome - a referred submission still gets a premium estimate at its risk rating.

Key Features

Submission form

Capture industry sector, employee count, annual revenue, claims in last 3 years, and location risk category for any SME applicant.

Rule-by-rule explanation

Every scoring rule is shown on the result page with its exact adjustment and the reason behind it. No black box - the underwriter can see exactly what drove the decision.

Decision output

Approved, Referred, or Declined with a risk rating and a revenue-adjusted premium estimate. Referrals get a suggested review note.

Portfolio dashboard

KPI cards for total submissions, approval rate, and average risk score. Charts for decision distribution and industry breakdown across the book.

Submissions log

Paginated list of all submissions with status badges, score, and drill-down to the full result for any prior application.

Seeded demo data

45 pre-seeded submissions across all risk bands and industries so the portfolio view is populated from the first load.

How It Would Work in Production

The current version takes manual form inputs. In a real insurer or MGA, the data pipeline would look different at almost every step:

  • Submission intake - broker portals (Acturis, Open GI) or direct digital channels would push structured submissions via API, replacing the manual form entirely for most volume
  • Claims data - rather than self-reported claims count, the engine would query the insurer's own claims system or market-level data (Lloyd's claims databases, IUA) to verify history automatically
  • Location risk - the four-tier manual input would be replaced by a postcode-level geodata lookup against flood risk (JBA Risk, Ambiental), subsidence, and crime indices - turning a subjective input into an objective one
  • Revenue and employee verification - Companies House filings and HMRC data to validate self-reported figures before they enter the scoring pipeline
  • Rating engine integration - the score and decision would feed into a downstream rating engine (Sequel Impact, Xuber) or policy admin system (Guidewire, Applied Epic) rather than generating a standalone output
  • Audit and compliance - every decision stored immutably with the full rule breakdown for FCA fair treatment review, complaint handling, and model governance

The rule pipeline architecture is deliberately built to support this: each rule is an independent function that takes a submission object and returns an adjustment with a reason. Replacing the manual location input with a real geodata API call means changing one function, not rebuilding the engine.

What I Learned

  • Building a Python FastAPI backend and deploying it as Vercel serverless functions alongside a React frontend - a different deployment model from Node.js
  • Designing a rule pipeline where explainability is a first-class requirement, not an afterthought - every rule must produce a human-readable reason, not just a number
  • The difference between a scoring model being correct and being trusted in a regulated context - audit trails, decision transparency, and referral logic matter as much as the score itself
  • SQLAlchemy with SQLite for a lightweight but proper ORM layer - good enough for a prototype, swappable for Postgres in production without touching the query logic
  • How premium calculation interacts with risk scoring in insurance: the rating band and revenue base are independent inputs to pricing, so a referred risk still gets a technically valid premium estimate
