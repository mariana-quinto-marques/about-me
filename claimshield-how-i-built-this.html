How I Built ClaimShield

An AI-powered motor insurance claims intelligence platform. Upload damage photos, get a Claude vision assessment. Open any claim and get an 8-signal fraud score, a signal-by-signal breakdown, and a recommended action - all explained, not just scored.

Try it live

The Idea

Insurance fraud costs the UK industry an estimated £1.1 billion a year, and motor claims are the highest-volume attack surface. Having shipped two insurance platforms at Azur Technology - including a financial lines and claims system - I'd seen how claims handlers actually work: PDFs attached to emails, fraud checks done manually against spreadsheets, decisions that are hard to audit and even harder to explain to a regulator.

The question I wanted to answer was: what does an AI-native claims tool look like - one where the fraud reasoning is transparent, the damage evidence is AI-assessed on upload, and the handler's job is to make the final call rather than do the analysis from scratch?

ClaimShield is that prototype. It's also the project where I first properly integrated the Anthropic Claude API rather than just using Claude Code as a build tool.

Tech Stack

This is the only project in this portfolio using Next.js and the only one making direct calls to the Anthropic Claude API. The Next.js API route is the right architectural choice here: it keeps the API key server-side and acts as a security boundary between the browser and the model.

The AI Integration

Two distinct Claude use cases in one app: multimodal vision for damage assessment and structured reasoning for fraud signal analysis. Different prompt designs, different output schemas, different jobs to be done.

Damage assessment via vision

The /api/assess route accepts 1-5 images as base64 data URIs and sends them to Claude as image blocks alongside a system prompt instructing it to act as an expert automotive damage assessor. The output schema is enforced in the prompt - Claude must return valid JSON, never markdown:

  • overallSeverity - minor / moderate / severe
  • damagedParts[] - part name, damage type, severity, repair vs. replace recommendation, estimated cost range
  • totalEstimatedCostRange - low and high bounds
  • claimNotes - professional paragraph suitable for the claim record

Images are resized client-side (canvas, max 1024px at 85% quality) before encoding to stay within API limits and reduce latency.

Fraud signal analysis

Each claim carries an 8-signal fraud assessment. Every signal has a source, a finding, a confidence percentage, and a verdict. The model also produces a prose reasoning summary explaining the overall score - not just what the signals found, but why they matter together:

Key Features

Overview dashboard

KPI cards (active claims, awaiting action, avg resolution time, fraud prevented), risk distribution donut, and a live activity feed.

Claims table

Filterable, searchable table across 10 columns with risk and status pill filters. Click any row to open the full claim detail panel.

Fraud hub

Flagged claims queue with fraud type breakdown bar chart and weekly volume spark chart. The view a fraud team lead would open first.

Claim detail panel

Full slide-over with policyholder, incident, narrative, vehicle, third party, evidence photos, timeline, fraud score ring, signal table, model reasoning, and handler notes.

AI damage assessment

Upload 1-5 photos and get a Claude vision assessment: damaged parts, severity, repair vs. replace recommendation, and a total cost estimate range.

Claim actions

Approve, Send for Review, Escalate to SIU, or Reject - with instant status updates and toast notifications. Status changes persist in app state.

How It Would Work in Production

The current version uses 10 mock UK claims and manual photo uploads. In a real insurer or MGA, almost every input would be automated:

  • Claims intake - structured FNOL (First Notice of Loss) data pushed from a claims management system (Guidewire ClaimCenter, Duck Creek) or a digital FNOL portal, replacing manual form entry
  • Policy velocity and duplicate detection - IFB (Insurance Fraud Bureau) database queries and cross-referencing against the insurer's own policy and claims history, rather than signal scores derived from mock data
  • DVLA/DVSA API - vehicle registration lookup to verify make, model, year, and MOT status automatically from the reg plate, flagging discrepancies with the claimant's description
  • Photo EXIF validation - stripping and analysing GPS coordinates and timestamps from uploaded photos against the reported incident location and time - a strong consistency signal that's hard to fake
  • Location verification - postcode geocoding against flood, crime, and accident hotspot data (Ordnance Survey, police.uk open data) to contextualise incident location risk
  • Repair estimate benchmarking - comparing submitted estimates against Audatex or Solera market data for the vehicle make/model/damage type to flag inflation
  • API security - the Next.js API route boundary stays; in production it would add request authentication, rate limiting, and a logging layer for model governance and FCA audit trail requirements

The Claude integration architecture is already production-ready in one key respect: the API key never touches the browser. The /api/assess route is the only place it's used, and it validates inputs before passing them to the model. Extending it to production would mean adding auth middleware and a request log, not restructuring the integration.

What I Learned

  • Prompting Claude for consistent structured JSON output from image analysis - including how to handle the optional markdown code fence wrapping Claude sometimes adds, and why parseAssessmentJSON() needs to strip it
  • Next.js API routes as the right security boundary for third-party API keys - calling the Anthropic SDK server-side means the key is never in the client bundle or network requests
  • Designing fraud signals where explainability is the product, not just the score - a claim handler needs to know why a claim scored 91, not just that it did
  • Client-side image resizing before API upload: canvas-based resize to max 1024px keeps API costs and latency manageable without sacrificing the detail Claude needs to assess damage
  • The difference between a fraud score being statistically accurate and being actionable - a signal that says "Review" with a clear finding is more useful to a handler than a flag with no context
